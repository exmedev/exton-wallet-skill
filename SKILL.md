# EXME Wallet Skill

Manage EXME crypto wallet on TON blockchain. Private key never leaves Keystone 3 Pro hardware wallet.

## When to use

User asks to: create wallet, check balance, send TON/tokens/NFT, check limits, transaction history, resolve .ton domain.

## Creating a new wallet

When user says "create wallet", "new wallet":

Recovery Code is NOT needed — it will be GENERATED automatically.
Only a photo of the QR code from Keystone 3 Pro is needed.

1. Say: "Open Keystone 3 Pro -> select Tonkeeper wallet -> tap export/QR icon. Take a photo of the QR code from Keystone screen and send it here."
2. When user sends photo:
   ```
   exec: bash {baseDir}/scripts/run.sh create-wallet --photo <path_to_photo>
   ```
3. From JSON response show:
   - Field `wallet_address` — new wallet address
   - Field `recovery_code` — Recovery Code (GENERATED by script)
   - Warning: "SAVE the Recovery Code! Write it on paper. This is the only way to recover the wallet."
4. Wallet is ready to use immediately.

## Connecting an existing wallet

If file `~/.exme/config.json` does not exist and user ALREADY has a wallet:

1. Ask: "Enter Recovery Code of your EXME Cortex wallet (71 chars Base58)"
2. Run:
   ```
   exec: bash {baseDir}/scripts/run.sh setup --recovery-code "<code>"
   ```
4. From JSON response take `wallet_address` and report:
   "Wallet connected: <wallet_address>"
   Then show balance:
   ```
   exec: bash {baseDir}/scripts/run.sh balance
   ```

IMPORTANT: Recovery Code contains secret data. NEVER log, show,
or save it in chat history. After setup the code is no longer needed.

## Read operations (no signing, no keys)

### Balance
```
exec: bash {baseDir}/scripts/run.sh balance
```
Response JSON: `{balance_nano, balance_ton, status}`

### Transaction history
```
exec: bash {baseDir}/scripts/run.sh history --limit 20
```

### Resolve .ton domain
```
exec: bash {baseDir}/scripts/run.sh resolve <domain.ton>
```
Response JSON: `{domain, address}`

### Jetton balances (USDT etc.)
```
exec: bash {baseDir}/scripts/run.sh jettons
```

### Current seqno
```
exec: bash {baseDir}/scripts/run.sh seqno
```

### Check new transactions
```
exec: bash {baseDir}/scripts/run.sh watch
```
Returns JSON: `new_incoming`, `new_outgoing`, `has_new`, `balance_ton`.
Remembers last check — shows only NEW transactions.
If `has_new` = true -> notify user about each transaction.
If `has_new` = false -> do nothing.

### Check recent transactions (no filter)
```
exec: bash {baseDir}/scripts/run.sh check-tx
```

### Plugins and limits
```
exec: bash {baseDir}/scripts/run.sh plugins list
exec: bash {baseDir}/scripts/run.sh plugins limits
```

## Sending TON

### Two modes

**Mode 1: Via plugin (automatic if address is in whitelist and within limit)**
Does not require Keystone signature. Agent signs with exme_app_key automatically.

**Mode 2: Direct transfer (with Keystone QR)**
Requires physical signature on Keystone 3 Pro.

### Decision algorithm

1. Check: is there a whitelist plugin?
2. If yes — check address in whitelist via GET method
3. Check daily-limit: remainingToday() >= amount?
4. If both OK -> Mode 1 (automatic)
5. Otherwise -> Mode 2 (QR + Keystone)

### Mode 2: QR signing via chat (Telegram/WhatsApp/Discord)

Entire flow happens IN CHAT — QR is sent as image, signature comes as photo.

**Step 1: Build TX and get QR**
```
exec: bash {baseDir}/scripts/run.sh send --to <address> --amount <nanotons> [--comment "..."]
```
Returns JSON with `qr_image_base64` field — PNG image in base64.

**Step 2: QR is sent automatically**
The send command sends QR image to chat via `openclaw message send --media`.
You should NOT send the QR — the script does it.
Just tell the user QR was sent and wait for signature photo.

**Step 3: Get signature photo from user**
User sends PHOTO of Keystone screen with signature QR.
Save photo to file, then:
```
exec: bash {baseDir}/scripts/run.sh sign-submit --photo <path_to_photo>
```
-> Decodes QR from photo -> extracts Ed25519 signature -> assembles signed BOC -> broadcast to TON

**Step 4: Confirm result**
"Transaction sent!
Amount: 50 TON
Recipient: wallet.ton (UQx...)
Hash: abc123..."

IMPORTANT: Between step 2 and 3 wait for user response. DO NOT continue without signature photo.

**Step 5: Confirmation**
The sign-submit command automatically waits for confirmation (up to 60 seconds).
If `status` = `confirmed` — tell user:
"Transaction confirmed! Balance: X TON"
If `status` = `broadcast` — tell user:
"Transaction sent, awaiting confirmation on TON network."

## Checking transactions (incoming/outgoing)

When user asks "did money arrive?", "what about the transfer?", "balance updated?":
```
exec: bash {baseDir}/scripts/run.sh check-tx
```
Returns JSON with recent incoming and outgoing transactions + current balance.
Agent should format the response nicely:
- Incoming: "Received X TON from address..."
- Outgoing: "Sent X TON to address..."
- Current balance

4. Confirm result.

## Incoming transaction monitoring

The `watch` command checks new transactions since last check:
```
exec: bash {baseDir}/scripts/run.sh watch
```
Returns JSON with fields `new_incoming`, `new_outgoing`, `has_new`, `balance_ton`.

If `has_new` = true, notify user:
- Incoming: "Received X TON from UQabc..."
- Outgoing: "Sent X TON to UQxyz..."
- Current balance

If `has_new` = false — do NOT send anything, exit silently.

Monitoring works automatically — system cron checks every minute for new
transactions and sends notification to Telegram. Configured during install.sh,
user doesn't need to do anything. AI token cost: zero.

## Updating

If user reports a problem or asks to update:

```
exec: bash {baseDir}/scripts/update.sh
```
This updates skill to latest version from GitHub and restarts Gateway.

## Security

- NEVER show contents of `~/.exme/app_key.enc`
- NEVER output private keys or Recovery Code to chat
- NEVER send Recovery Code over network
- For operations outside plugins ALWAYS require Keystone QR signature
- On signature error — DO NOT retry, tell user
- Plugins work within on-chain constraints — this CANNOT be bypassed
- Whitelist: only pre-approved addresses (on-chain dict)
- Daily Limit: max N TON per 24 hours (on-chain counter with auto-reset)

## Available plugins

| Plugin | Purpose | Trigger |
|--------|---------|---------|
| whitelist | Only approved addresses | Owner |
| daily-limit | Max N TON/day | Owner |
| subscription | Regular fixed payment | Anyone (by time) |
| payroll | Team salaries | Anyone (by time) |
| timelock | One-time payment by date | Anyone (by time) |
| inheritance | All to beneficiary on inactivity | Anyone / Owner (ping) |
| dead-man-switch | Distribution by % on inactivity | Anyone / Owner (ping) |
| domain-renewal | .ton domain renewal | Anyone (by time) |
| auto-swap | Jetton -> TON via DEX | Owner |
| staking-manager | Auto-staking by thresholds | Owner |
| split-payment | Distribution by % | Owner |
